The Ark Challenge - skipinx

We're given the following files.

├── docker-compose.yml
├── nginx
│   ├── Dockerfile
│   └── default.conf
└── web
    ├── Dockerfile
    ├── index.js
    ├── package-lock.json
    └── package.json

3 directories, 7 files

As you can tell from the files, the set up is simply an Nginx proxy in front of a NodeJS application. Looking into the default.conf Nginx config we see the following code.

server {
  listen 8080 default_server;
  server_name nginx;

  location / {
    set $args "${args}&proxy=nginx";
    proxy_pass http://web:3000;
  }
}

This seems very simple, it's just adding a proxy query parameter onto any request. The following excerpt shows the code of the NodeJS server, and we get a bit more context on what this proxy query parameter is doing.

const app = require("express")();

const FLAG = process.env.FLAG ?? "SECCON{dummy}";
const PORT = 3000;

app.get("/", (req, res) => {
  req.query.proxy.includes("nginx")
    ? res.status(400).send(JSON.stringify(req.query))
    : res.send(`Congratz! You got a flag: ${FLAG}`);
});

app.listen({ port: PORT, host: "0.0.0.0" }, () => {
  console.log(`Server listening at ${PORT}`);
});

It's incredibly simple, if the NodeJS server detects that the proxy query parameter contains nginx, then it will not return the flag. The challenge objective is pretty straight forward then, either change or remove the proxy query parameter so that it doesn't contain nginx.

It looks like req.query.proxy was undefined! This means that Nginx failed to set the proxy query parameter. In reality, Nginx had actually updated the query parameters to include proxy=nginx but we were hitting a default ExpressJS parameter limit. This limit is mentioned in the README.md of the qs library used by ExpressJS for query parameter parsing:

"For similar reasons, by default qs will only parse up to 1000 parameters."

We can see the implementation of that limit here: /lib/parse.js L66-74 | qs. Note that there is logic to throw a RangeError if the limit is exceeded, which would prevent our exploit, but this error is only thrown when the throwOnLimitExceeded option is true. It is false by default, as defined here: /lib/parse.js L29 | qs.

This means that we can send 999 test= query parameters and a single proxy=ourvalue parameter so that when the Nginx proxy adds proxy=nginx it is simply ignored by ExpressJS.

Request

GET /?test=&test=[...snip...]proxy=test HTTP/1.1
Host: localhost:8000

Response

HTTP/1.1 200 OK
...snip...
Congratz! You got a flag: SECCON{dummydummy}
